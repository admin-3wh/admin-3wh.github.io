Triager+: From Concept to Execution

Written by 3wh Admin · Published on Jul 26, 2025

Triager+ began as a challenge — to combine machine learning with a usable, focused web interface for classifying help desk tickets. But it evolved into something more: an exercise in product minimalism, memory-aware architecture, and end-to-end delivery from backend to UI. This is the story of its construction.

What Is Triager+?

Triager+ is a browser-based tool that accepts unstructured help desk ticket descriptions and classifies them into predefined categories and priorities. It supports two models: a lightweight Naive Bayes and a (currently disabled) DistilBERT deep learning alternative.

Why Build It?

The goal was to create something useful but deployable, intelligent but minimal. No extra packages, no large dependencies if avoidable. It was also a technical portfolio project — to show you can build something real, not just theoretical.

Architecture Overview

The system is composed of:

  • A FastAPI backend that handles requests, preprocessing, and model inference
  • scikit-learn for training and saving the Naive Bayes models
  • A client-side interface using HTML/CSS/JavaScript
  • Hosted via Render (backend) and GitHub Pages (frontend)

Training the Models

The Naive Bayes model was trained on labeled ticket data using TF-IDF vectorization. It predicts both category and priority. DistilBERT was finetuned but disabled due to RAM constraints on free Render plans.

Handling Constraints

The project faced real-world constraints:

  • CORS errors requiring careful middleware setup
  • Memory limits leading to the decision to gracefully disable the transformer model
  • Latency optimization for fast response on low-cost infrastructure

Designing the UI

UI design was kept intentionally minimal. Buttons, confidence percentages, a clear reset option, and basic mobile responsiveness were all tuned by hand. An optional analytics system (Google Analytics) was added with future usage tracking in mind.

Lessons Learned

  • Small projects still require deployment discipline
  • Frontend error handling needs to anticipate every backend fail mode
  • Tools that say little but do something useful will always have value

What’s Next?

Triager+ is complete — but the methodology behind it (quick experiments, direct feedback, real-world context) will drive future work. A second project is already in motion: ShapeSound.
